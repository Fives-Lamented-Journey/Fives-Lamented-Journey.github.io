在 Linux 系统中，**中断处理被拆分为上半部分（Top Half）和下半部分（Bottom Half）**，核心目的是**缩短中断关闭的时间**，保证系统的实时性和吞吐量。

### 核心设计思想

中断发生时，CPU 需要立即响应硬件请求，但如果中断处理函数执行时间过长，会导致：

1. 其他中断被长时间屏蔽，引发硬件响应延迟。

2. CPU 被中断处理占用，无法处理进程调度等核心任务。

因此 Linux 将中断处理拆分为两个阶段：

- **上半部分**：**快速执行、不可中断**，只做最核心的硬件操作。

- **下半部分**：**延迟执行、可中断**，处理耗时的后续逻辑。

---

### 一、 中断上半部分（Top Half）

这是中断发生时**立即执行的处理程序**，对应内核中的 `request_irq()` 注册的中断服务函数（ISR）。

#### 1. 核心特点

| 特性 | 说明 |
| --- | --- |
|**执行时机**|中断触发后立刻执行，抢占所有进程|
|**执行状态**|关中断（或关当前中断线），**不可被打断**|
|**执行时长**|必须**极短**，通常几微秒到几十微秒|
|**功能范围**|只做最必要的硬件操作|

#### 2. 典型操作

1. **确认中断源**：告知硬件“已收到中断信号”，避免硬件重复触发。

2. **读取硬件数据**：将硬件缓冲区的数据拷贝到内核内存。

3. **标记下半部分任务**：通过标记、队列等方式，通知下半部分后续处理。

4. **清除中断标志位**：让硬件可以触发下一次中断。

#### 3. 示例逻辑

```C

// 上半部分中断处理函数
irqreturn_t my_isr(int irq, void *dev_id) {
    struct my_device *dev = dev_id;
    
    // 1. 读取硬件寄存器数据
    dev->data = readl(dev->reg_base + DATA_REG);
    // 2. 清除硬件中断标志
    writel(0x01, dev->reg_base + INT_CLEAR_REG);
    // 3. 标记下半部分需要处理
    schedule_work(&dev->work);  // 触发 workqueue 下半部分
    
    return IRQ_HANDLED;
}
```

---

### 二、 中断下半部分（Bottom Half）

这是**延迟执行的处理逻辑**，在上半部分退出后，由内核调度在合适的时机执行。

#### 1. 核心特点

| 特性 | 说明 |
| --- | --- |
| **执行时机** |上半部分完成后，由内核调度（如软中断、线程）|
|**执行状态**|开中断，**可被更高优先级中断抢占**|
|**执行时长**|可以较长，允许处理复杂逻辑|
|**功能范围**|处理耗时的业务逻辑|

#### 2. Linux 中常用的下半部分实现机制

Linux 内核提供了多种下半部分机制，不同版本有不同的主推方案，核心演进路线：`BH` → `tasklet` → `workqueue`。

|机制|核心特点|适用场景|
|---|---|---|
|**软中断（Softirq）**|静态分配，CPU 绑定，执行效率最高|高频率、低延迟场景（如网络收发、块设备 I/O）|
|**Tasklet**|基于软中断实现，动态注册，支持禁用/启用|中等频率的中断后续处理（如串口数据处理）|
|**工作队列（Workqueue）**|基于内核线程实现，可睡眠|耗时较长、需要睡眠的场景（如磁盘文件写入）|

#### 3. 典型操作

1. **数据处理**：解析上半部分读取的硬件数据。

2. **数据转发**：将数据拷贝到用户空间，或转发给其他内核模块。

3. **状态更新**：更新设备状态、统计信息等。

4. **唤醒用户进程**：通知等待数据的用户态进程。

---

### 三、 上半部分与下半部分的协作流程

以**网卡接收数据**为例，完整流程如下：

1. 网卡收到数据，向 CPU 发送中断请求。

2. CPU 暂停当前任务，**关中断**，执行网卡中断上半部分：

    - 确认中断，读取网卡缓冲区数据到内核内存。

    - 清除网卡中断标志，触发软中断（`NET_RX_SOFTIRQ`）。

    - 开中断，返回被中断的任务。

3. 内核调度软中断执行（下半部分）：

    - 解析网络数据包，执行协议栈逻辑（如 TCP/UDP 解包）。

    - 将数据拷贝到用户态应用程序的缓冲区。

    - 唤醒等待数据的应用程序进程。

---

### 四、 关键区别总结

|对比维度|上半部分|下半部分|
|---|---|---|
|执行优先级|最高（抢占所有进程）|低于硬件中断，高于普通进程|
|中断状态|关中断（不可抢占）|开中断（可被硬件中断抢占）|
|执行时长|必须极短|可较长|
|睡眠能力|绝对不允许睡眠|部分机制允许睡眠（如 workqueue）|
|触发方式|硬件中断触发|上半部分主动触发|

---

### 五、 内核编程中的注意事项

1. **上半部分严禁睡眠**：上半部分执行时关中断，睡眠会导致系统死锁。

2. **数据共享需加锁**：上半部分和下半部分共享数据时，必须使用自旋锁（`spinlock`）。

3. **选择合适的下半部分机制**：高频低延迟用软中断/tasklet，耗时任务用 workqueue。
